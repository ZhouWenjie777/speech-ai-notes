1. 多态
	- 多态（polymorphism）就是“同一接口，多种实现”——用基类的指针/引用去操作对象，实际执行的行为却由“真正指向的派生类类型”决定
	- 指针是基类脸，落地却跑派生类代码
	- 编译期多态（静态多态）
		- 模板 + 重载：函数名一样，编译器根据实参类型当场决定调哪个版本
		- `void print(int); void print(double);   // 重载`
		- `template<class T> void foo(T t){}      // 模板`
		- 优点：零运行时开销；缺点：代码膨胀，类型必须在编译期已知
	- 运行期多态（动态多态，OO（Object-Oriented） 常说的“多态”专指这个）
		- 通过 virtual 函数 + 基类指针/引用 实现
		- `struct Shape { virtual double area() const = 0; };   // 抽象接口`
		- `struct Circle : Shape { double r; double area() const override { return 3.14*r*r; } };`
		- `struct Rect : Shape { double a,b; double area() const override { return a*b; } };`
		- `void show(const Shape& s){ std::cout << s.area(); }  // 同一接口`
		- `Circle c{1}; Rect r{2,3};`   // [[聚合初始化]]
		- `show(c); show(r);    // 多种实现`
2. 单继承与派生类
	- 基类（`TableTennisPlayer`）：
		- 已存在的类，封装共性数据与行为；派生类通过“继承”复用其代码，并扩展新功能。
	- 派生声明格式
		- `class 派生类名 : 派生方式 基类名 { ... };`
		- 例：`class RatedPlayer : public TableTennisPlayer { ... };`
		- 公有派生（public）保持基类公有接口对外可见，是“is-a”关系最常用方式
	- 构造函数责任链
		- 派生类构造函数必须先调用基类构造函数（显式或隐式），再初始化自身新增成员；
		- 写法：`RatedPlayer(unsigned int r, const string &fn, const string &ln, bool ht) : TableTennisPlayer(fn, ln, ht), rating(r) { }`
		- 也可用已有基类对象复用构造：`RatedPlayer(unsigned int r, const TableTennisPlayer &tp) : TableTennisPlayer(tp), rating(r) { }`
	- 访问规则
		- 派生类成员函数不能直接访问基类 private 成员（如 `firstname、lastname、hasTable`），只能通过基类提供的 public/protected 接口使用；若基类将数据设为 protected，则派生类可直接访问
	- 名字隐藏与重载
		- 派生类若定义与基类同名函数（即使参数不同），会隐藏基类所有重载版本；想暴露基类版本需写 `using Base::Func;`
	- 内存布局（单继承）
		- 派生对象 = “基类子对象” + “新增成员”；基类指针可自然指向派生对象，实现多态基础
	- 析构函数
		- 若基类析构非 virtual，通过基类指针删除派生对象会导致未定义行为；后续引入 virtual 可正确调用派生析构
	- 示例调用
		- `RatedPlayer rp(1200, "Li", "Na", true);`
		- `rp.Name();           // 继承自基类`
		- `cout << rp.Rating(); // 派生类扩展接口`
3. 多态性的使用（公有继承 + 虚函数）
	- 多态前提
		- 公有继承：派生类对象“is-a”基类，可用基类指针/引用管理
		- 虚函数：基类把可能变化的行为声明为 `virtual`，派生类 `override` 提供专属实现
	- 运行流程
		- 基类指针 `Brass* bp` 指向 `BrassPlus` 对象时，调用虚函数 `bp->ViewAcct()`
		- 先到派生类找同名函数，存在则执行派生版本
		- 不存在，沿继承链向上回溯，执行最近祖先的虚函数实现
	- 编码要点
		- 遍历容器（基类指针数组/`vector<unique_ptr<Brass>>`）即可统一处理“所有账户”，无需区分具体类型
		- 构造函数**不能**是虚函数；析构函数**建议**虚函数，否则通过基类指针 `delete` 派生对象会**切片析构**，造成资源泄漏
		- 派生类重写时函数签名必须与基类完全一致（C++11 用 `override` 关键字让编译器帮忙检查）
	- 示例效果
		- 同一行 `bp->ViewAcct();`
			- 若 `*bp` 实际是 `Brass` → 输出基本账户信息
			- 若 `*bp` 实际是 `BrassPlus` → 额外显示透支额度、欠款、利率
		- 客户端代码零改动，扩展新账户类型时只需新增派生类并重写虚函数即可
	- 析构不是 virtual：切片析构
		- 基类析构函数不是 virtual， delete 基类指针时只会执行基类析构函数
			- 基类子对象里的动态内存被释放
			- 派生类子对象（包括它申请的堆内存）完全不会被析构，造成资源泄漏
		- 把基类析构函数声明为`virtual`后，运行时根据实际对象类型先调派生类析构，再调基类析构，整条链的资源都能正确释放
	- 继承不是 public：就没有“基类指针指向派生对象”这条通路
		- 指针/引用转换被切断
			- `Derived d; Base* pb = &d;  // 编译错误`，因为`private/protected`继承后，编译器认为`Derived`不是`Base`，外部拿不到基类指针/引用，自然无法通过基类接口调用虚函数
		- 即使强转也调不到派生实现
			- 手动`static_cast`能骗过编译，但运行期依旧不会走派生类虚函数表，还是基类行为——多态机制被语言规则屏蔽
	- 构造不能是 virtual：简单、安全、无歧义
		- 虚函数靠 `vptr → vtable`
			- 每个带虚函数的类，对象里藏一个隐藏指针 `vptr`，指向类的虚函数表
			- `vptr` 是在构造函数执行期间才由编译器写入对象的，对象还没出生，就没有 `vptr`，所以“虚构造函数”无从查表，也毫无意义
		- 语言用途
			- “虚”是为了通过“基类指针/引用”实现运行期多态
			- 而创建对象时是显式写出具体类型，编译器早已知道该调谁的构造，根本不需要动态分派
