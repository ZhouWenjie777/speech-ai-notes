# 1. 编译器和解释器
1. 影响计算性能的主要因素：命令式编程、符号编程、异步计算、自动并行和多GPU计算
2. 命令式编程瓶颈
	- Python 按顺序执行 `e=add(a,b)`、`f=add(c,d)`、`g=add(e,f)`，每一步都存中间变量、调解释器，GPU 上开销巨大；无法跨调用做融合或并行，效率受限。
3. 符号式编程
	- 先定义完整计算图再编译执行，可跳过Python解释器，消除CPU-GPU瓶颈。
	- 编译期见全局代码，可常量折叠、内存复用、算子融合，甚至将图导出为与Python无关的格式供其他环境运行，效率与可移植性均优于命令式。
	- 常量折叠（constant folding）
		- 编译器的一种优化技术，在编译阶段就把纯常量表达式直接算出结果，避免运行时重复计算，节省指令与寄存器。
		- 比如把 `print(((1+2)+(3+4)))` 直接折成 `print(10)` ，减少内存与计算
	- 内存复用（memory reuse）
		- 编译器在静态分析后，把生命周期不重叠的中间张量共享同一块物理内存
		- 符号式图编译时能看见完整数据流，若张量 A 后续不再被使用，而张量 B 即将诞生，编译器可把 B 直接放到 A 的内存地址上，运行时不额外分配新buffer，显存/内存峰值下降
		- 与命令式“每步都保留中间变量”相比，可显著减少深度学习模型的内存占用
	- 算子融合（operator fusion）
		- 把多个连续的小算子合并成一个更大的算子，在同一块内存上一次性完成计算，减少访存（显存读写）和启动（内核启动）开销
		- 符号式编程在编译期就能看到完整算子链，因此可以自动做这类融合优化；命令式逐条执行，无法实现
4. 混合式编程
	- PyTorch 动态图保持开发灵活，通过 `torch.jit.script` 把模型编译成静态计算图（TorchScript），一次性优化并导出非 Python 格式；同一代码既可调试又可在 多卡（如8-GPU） 服务器上并行执行，消除单线程 Python 瓶颈，兼顾效率与可移植性。
	- 开发阶段
		- PyTorch 动态图开发模型，逐行调试、打印中间值
	- 训练/部署阶段
		- 通过 `torch.jit.script(net)` 把模型编译成静态图，该图被优化（算子融合、内存复用、常量折叠），导出为完全脱离 Python 的二进制，从而不再受 Python GIL 限制能够并行。
		- 跨卡数据并行+卡内算子并行，两者都因绕过Python GIL而可以同时全速运行
	- Python GIL（Global Interpreter Lock）
		- Python 解释器的一把“全局锁”，让同一时刻只能有一个线程执行 Python 字节码，所以纯 Python 代码在多核 CPU 上无法真正并行
		- TorchScript 把计算图编译成不依赖 Python 解释器的原生程序，于是可以绕过这把锁
# 2. 异步计算
1. 异步编程与设备并行
	- Python 单线程本质不变，框架把 GPU 操作变成**异步队列**：调用立即返回，内核在后台排队执行，CPU 可继续发下一批任务；跨 CPU、多 GPU 与同一 GPU 内部计算并发运行，减少空闲、提升吞吐并降低内存峰值。
2. 后端异步处理
	- PyTorch GPU 操作默认异步：前端把内核排入后端队列立即返回，Python 继续执行；未同步时计时器只测“排队时间”，故显似“快几个数量级”。强制 `torch.cuda.synchronize(device)` 等待后端完成，暴露 GPU 并行矩阵乘的高速度。
3. 前端与后端交互
	- Python 前端只把算子塞进后端队列立即返回；后端 C++ 线程按依赖图调度执行。前端仅在取结果（如 `print(z)`）时阻塞等待，因而 Python 性能几乎不影响整体速度，依赖图保证相互依赖的操作顺序执行，无数据竞争。
4. 并发基础速记
	- 在GPU中，‌Warp 是硬件层面的最小执行单元，是指线程调度和执行的基本单位，通常由固定数量的线程（如32个）组成（遵循SIMT模型，即单指令多线程）
	- OS 调度（Operating System Scheduling）就是操作系统把线程/进程分配给CPU 核上运行的过程

| 概念                  | 厨房比喻   | 专业一句话                                          | 现状                                     |
| ------------------- | ------ | ---------------------------------------------- | -------------------------------------- |
| **进程**              | 独立餐厅   | 资源隔离的地址空间，OS 调度的最小单位                           | 开多进程=开多家店，成本大                          |
| **线程**              | 餐厅厨师   | 共享地址空间的执行流，CPU 调度的最小单位                         | 创建快，切换轻                                |
| **CPU 物理核**         | 灶台炉头   | 真正并行执行指令的硬件单元                                  | 主流 CPU 4-16 个                          |
| **CPU 超线程(SMT/HT)** | 一炉双锅   | 一个物理核维护两套架构状态，分时执行两条指令流                        | 现在笔记本/台式几乎标配，每核 2 逻辑线程                 |
| **GPU core**        | 大排档小炉头 | 单 GPU 含数千 core，被同一 warp-scheduler 驱动，同步执行同一条指令 | 1 warp = 32 线程，数据并行，单线程无法独占 core 跑不同指令 |
5. 小节
	- 前端只排队，后端并行执行，Python 零阻塞；但队列过长会爆内存，建议每小批量同步一次
	- 芯片厂商（NVIDIA Nsight、Intel VTune）提供性能分析器，可精确查看内核耗时、带宽与占用率
# 3. 自动并行
1. 自动并行要点
	- 深度学习框架后端自动建图，识别无依赖节点并行执行；单操作符（如 `dot`）已占满单设备 CPU/GPU 资源，本地并行收益有限  
	- 多设备场景关键：跨 GPU 数据并行 + CPU 辅助计算，框架用几行 Python 即可自动调度，实现计算与通信重叠
2.  多GPU并行示例
	- 分别预热后顺序跑：GPU1 0.51 s、GPU2 0.50 s，总和≈1 s
	- 去掉中间同步，让框架自动并行，总时间 0.58 s，小于两者相加
	- 框架自动把两设备计算重叠，无需复杂代码即享并行加速。
3. 计算与通信重叠
	- 默认先算完再复制：GPU 运行 0.51 s → CPU 复制 2.44 s，总 2.95 s
	- `to()`和`copy_()`等函数都允许显式的non_blocking参数，允许绕过同步
	- 设`non_blocking=True`，系统边算边用 PCIe 传数，总时间降至 1.88 s
	- 框架自动在“GPU 计算”与“CPU-GPU 通信”之间流水线并行，无需手动调度。
4. 小结
	- 系统拥有多GPU、多CPU及PCIe、SSD、网络等并行资源
	- 后端依计算图依赖自动把“计算-通信-复制”流水线化，手动调度痛苦，框架一键实现并行与重叠，最大化吞吐。
# 4. 硬件
1. 计算机组件与带宽
	- CPU：8+ 核，主控；内存直连，~100 GB/s；PCIe 4.0 ×64 通道，每通道 16 Gb/s 双向，通道总带宽 2×64×16 Gb/s ≈ 256 GB/s，连接 GPU、SSD、网卡
	- GPU：经 PCIe 插卡，单卡带宽 32 GB/s（×16），算力高，内存带宽 0.5-1 TB/s
	- SSD/NVMe：走 PCIe，顺序3-7 GB/s，随机IOPS高，持久存储训练数据与检查点
	- 网络：1-100 Gb/s 以太网 / InfiniBand，多机梯度同步，延迟 50 μs-1 ms
	- 带宽链：持久 → 内存 → CPU/GPU → 网络，任何环节慢都会让处理器空等，需通信和计算交错、预取和批量传输来隐藏延迟
2. 内存访问要点
	- CPU的内存通常为DDR4类型
	- DDR4 单模块 20–25 Gb/s，双通道峰值 40–100 Gb/s；首地址传输约 100 ns，后续突发 0.2 ns，首读成本 500×，应连续存取、对齐 64 位并利用多bank 4×并发
	- GPU 用更宽总线（352 bit RTX2080Ti）与高频 GDDR6，带宽 500+ Gb/s，容量小、价高，仅为高吞吐设计，技术细节对调参透明。
3. 存储层级
	- HDD：16 TB 便宜，8 ms 寻道 ≈ 100 IOPS，带宽 100–200 MB/s，适合归档
	- SSD: 10–50 万 IOPS，1–3 GB/s；块擦写（256 KB）致随机写慢、寿命有限， wear-leveling 分摊磨损
	- NVMe-SSD：走 PCIe，4 通道 PCIe 4.0 理论 8 GB/s，延迟低，训练数据首选
	- 云盘：IOPS与带宽可按需配置，小记录过多时升配 IOPS 降延迟
4. CPU 概览
	- 中央处理器（central processing unit，CPU）是任何计算机的核心
	- 处理器核心 （processor cores）用于执行机器代码
	- 总线（bus）用于连接不同组件
	- 缓存（cache）相比主内存实现更高的读取带宽和更低的延迟内存访问
	- 现代 CPU 含 AVX-512 向量单元，加速线性代数与卷积
	- 四核 Skylake 例：环总线连接核心与 64 MB L3、集成 GPU；每核私 L1/L2，共享 L3 提供高带宽低延迟。PCIe、USB、SSD 控制器等外设直连 CPU 或芯片组
	- 微体系结构
		- 前端取指+分支预测→解码成微指令→重命名→派遣到 8+ 并行执行单元（整型、FP/ASIMD、Load/Store）
		- 只要指令独立，每周期可发多条；分支预测错误会刷新流水线，故预测精度直接影响吞吐
	- 矢量化
		- ARM NEON / x86 AVX2 提供 128-512 位 SIMD 寄存器，单周期可对 8-64 对数做并行加/乘； fused-multiply-add 把乘加合并一条指令。
		- 服务器 CPU 借 OpenVino 等框架可获高吞吐，但 RTX2080Ti 的 4352 CUDA core 每周期同样执行 FMA，数量级优势仍碾压单 CPU。
	- 缓存层次
		- 寄存器：零延迟，数十个，编译器/程序员管理；
		- L1 32–64 KB 分 I/D，命中 ns 级；L2 256–512 KB/核，独占或共享；L3 4–256 MB 多核共享 
		- 向前遍历、局部访问提升命中率；缓存未命中代价高，**错误共享**（跨核同 cache line 写）致双方等待，多核反而更慢，决定缓存尺寸需权衡面积与一致性开销
5. GPU 与加速卡
	- 训练卡：需存全部中间激活求梯度，需大显存高带宽 HBM2/GDDR6 及至少 FP16 或 FP16/FP32 混合精度
	- 推断卡：仅前向，可 INT8/FP16，显存省、算力要求低
	- NVIDIA Turing T4 定位推断，V100 定位训练，分别优化内存与计算规模。
	- Turing SM 含 16 INT32 + 16 FP32 + 2 TensorCore/块，4 块组一个 SM，12 SM 再组 GPC，模块化设计可裁剪缺陷块提升良率。
	- TensorCore 专精 4×4~16×16 矩阵乘加，单周期 FMA 吞吐量数倍于传统 ALU，框架层自动调用，对用户透明。
6. 网络和总线/互连方式
	- PCIe：
		- ⼀种专用总线，用于每个通道点到点连接的高带宽需求。
		- 4.0×16 单向 32 GB/s，延迟~5 μs，优先大块传输，限 GPU 数量  
	- 以太网：
		- 连接计算机最常用的方式，比PCIe慢得多
		- 1 Gb/s~100 Gb/s，低成本长距，云实例常见 25/100 G，适合机架级聚合  
	- 交换机：
		- 一种连接多个设备的方式，任何一对设备可以同时执行（通常是全带宽）点对点连接。
		- 以太网 40 口或 PCIe 交换（P2 实例），扩展 GPU 池  
	- NVLink：
		- PCIe的替代品，适用于非常高带宽的互连
		- 300 Gbit/s/链路，V100 六链路，RTX 单链路 100 Gbit/s，用 NCCL 实现 GPU 间高带宽 P2P，替代 PCIe 瓶颈
7. 硬件小结
	- 传输量大次少：RAM/SSD/网络/GPU 皆然，减少往返开销  
	- 向量化匹配加速器：Xeon INT8、Volta FP16、Turing FP16/INT8/INT4，训练防溢出，推断可量化  
	- 内存对齐：64 位 CPU 按 8 字节，GPU 卷积尺寸对齐张量核，避免数据混叠导致性能退化  
	- 算法先画纸面性能：关注数量级，缓存命中可获 10×+ 加速；用调试器找瓶颈  
	- 训练 vs 推断硬件：训练重显存带宽，推断重算力单价，按需选型
# 5. 多GPU训练
1. 多GPU拆分策略
	- 网络拆分PP：
		- 每层放不同GPU，显存分段，但层间带宽需求高、同步复杂，不推荐
	- 层内拆分TP：
		- 通道/输出单元分卡，系统总显存随卡线性增加，单卡显存占用线性减少，需屏障且通信量大，也不推荐  
	- 数据并行DP：
		- 各GPU持完整网络，独立、同步维护模型参数更新，仅样本分片，前向反向后聚合梯度，同步简单、批量随卡数线性增，显存足时首选
2. 数据并行训练流程
	- 小批量随机划分为 k 份，均匀分发到 k 个 GPU
	- 各 GPU 前向+反向，得到本地梯度
	- 所有梯度 AllReduce 平均，得到全局小批量梯度
	- 同一梯度广播回各 GPU，各自更新完整参数
3. 小结
	- 拆分策略：层间/层内需复杂数据编排，数据并行最简单  
	- 数据并行：各 GPU 用相同模型、不同样本前向+反向，梯度 AllReduce 平均后广播，实现简单、扩展性好  
	- 有效批量 = k × 单 GPU 批量，需同步提高学习率；BatchNorm 改为 per-GPU 或同步 BN，保持统计一致
# 6. 多GPU的简洁实现
```python
from torch import nn
def train(net, num_gpus, batch_size, lr): 
	train_iter, test_iter= d2l.load_data_fashion_mnist(batch_size) 
	devices = [d2l.try_gpu(i) for i in range(num_gpus)]
	
	def init_weights(m): 
		if type(m) in [nn.Linear, nn.Conv2d]:
			nn.init.normal_(m.weight, std=0.01) 
	net.apply(init_weights) 

	# 在多个GPU上设置模型 
	net = nn.DataParallel(net, device_ids=devices) 
	trainer = torch.optim.SGD(net.parameters(), lr) 
	loss = nn.CrossEntropyLoss()
	timer, num_epochs = d2l.Timer(), 10
	animator = d2l.Animator('epoch', 'testacc', xlim=[1, num_epochs]) 
	for epoch in range(num_epochs): 
		net.train() 
		timer.start() 
		for X, y in train_iter:
			trainer.zero_grad()
			X, y = X.to(devices[0]), y.to(devices[0])
			l = loss(net(X), y)
			l.backward()
			trainer.step()
		timer.stop()
		animator.add(epoch+1, (d2l.evaluate_accuracy_gpu(net, test_iter),))
	print(f'测试精度：{animator.Y[0][-1]:.2f}，{timer.avg():.1f}秒/轮，'
		  f'在{str(devices)}')
```
1. 流程（`DataParallel` 内部）：
	- 用户代码只和主卡打交道（默认 `devices[0]`）, `DataParallel` 会在 `forward` 里自动按 batch 维度拆成数据，分别送到 `devices` 列表里的每张卡
	- 正向：输入按 batch 维度切 → 各卡并行 forward
	- 反向：各卡分别计算梯度 → 梯度汇总到主卡（底层自动All-Reduce）
	- 主卡用汇总后的梯度执行一次优化器 `step()` ，得到新参数
	- 主卡通过 `Broadcast` 把新参数整体复制到其余卡
# 7. 参数服务器
1. 分布式梯度聚合代价
	- Hierarchical All-Reduce（层级全归约）
		- 星型到主卡：3×160 MB /16 GB/s ≈ 30 ms + 回传 30 ms → 60 ms  
	- Parameter Server（参数服务器）
		- 全到 CPU：4×160 MB /16 GB/s ≈ 40 ms + 回传 40 ms → 80 ms  
	- Ring All-Reduce / Tree All-Reduce
		- PCIe 交换全带宽并行，带宽全利用，延迟最低
		- 分片4×40 MB，2.5 ms ×3 跳 + 回传 7.5 ms → 15 ms
	- 策略差异：拓扑与分片策略决定同量 160 MB 同步可在 15 ms–80 ms 间变化，设计时需先画带宽图再选算法
2. 环 All-Reduce 思维实验
	- n 节点环，整片传递需 n−1 步，仅单节点通信
	- 改将梯度切 n 块，每节点同步 1/n 大小，时间 ≈ (n−1)/n → 1（与 n 无关）
	- 实例：8×V100 同步 160 MB，2×160 MB÷(3×18 GB/s)≈6 ms，优于使用PCIe链路连接到主机CPU；若框架能把小块合并成大突发，实际性能更佳。
3. 多机训练步骤与瓶颈
	- 每机读不同批量，CPU→GPU 数据并行前向得本地梯度  
	- 机内 GPU 梯度聚合到本地主卡或分片  
	- 本地 CPU 将梯度推送到中央参数服务器（PS）  
	- PS 聚合全局梯度，更新参数并广播回各机 CPU  
	- 机内CPU→GPU 下发新参数，继续迭代
	- 瓶颈：中央 PS 带宽固定，m 工作节点推梯度耗时 O(m)；增 PS 至 n 台，每台存 O(1/n) 参数，更新耗时 O(m/n)，匹配后总时间 O(1)，实现线性伸缩。实际同一机器兼工作节点与服务器，需精细调同步，避免中央聚合成为分布式瓶颈。
4. 键值存储抽象
	- 将梯度归约视为交换操作 $g_i = \sum_{k\in workers}\sum_{j \in GPUs} g_{ijk}$，顺序无关；用键 $i$ 索引每层梯度，提供
		- **push(key, value)**：把梯度从工作节点推入公共存储聚合
		- **pull(key)**：取回聚合梯度用于更新
	- 隐藏同步复杂度，建模人员只需调用 push/pull，系统工程师在后台实现分布式聚合与一致性。
