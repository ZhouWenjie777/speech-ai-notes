1. 指令微调（Instruction Tuning）
	- 是指使用自然语言形式的数据对预训练后的大语言模型进行参数微调
	- 也被称为有监督微调（Supervised Fine-tuning）或多任务提示训练（Multitask Prompted Training）
2. 指令数据的构建
	- （1）指令数据示例
		- | 组成 | 说明 | 示例片段 |
		  |---|---|---|
		  | 任务描述（指令） | 告诉模型该做什么，零样本场景必备 | “将下列中文句子翻译成英文：” |
		  | 任务输入 | 待处理文本 | “今天天气很好。” |
		  | 任务输出 | 期望回答 | “The weather is nice today.” |
		  | 可选示例 | 上下文 few-shot 演示 | “例：中文：你好 → 英文：Hello。现在请翻译：今天天气很好。” |
	- （2）基于现有的NLP任务数据集构建
		- 给NLP任务的“输入-输出”对数据添加任务描述信息，即得到指令数据
			- | 原始任务 | 输入 | 输出 | 翻转 |
			  |---|---|---|---|
			  | 中英翻译 | 大模型已成 ML 重要方向 | Large language models... | 英→中 + 指令“请翻译成中文” |
			  | NLI | 前提：小孩在公园；假设：他在室外 | 蕴含 | 指令改为“请判断假设是否被前提蕴含” |
			  | QA | 法国首都是？ | 巴黎 | 指令“请根据答案生成问题：巴黎”→新问题样本 |
			- 翻转技巧：输入-输出对调 → 自动生成逆向任务（摘要↔生成、问题↔答案），一份标注两倍指令，提升多样性零成本
		- 关键工具
			- `PromptSource`：众包模板平台，支持多人共建/审校任务描述，已内置 100+ 数据集模板
			- FLAN/P3/Super-Natural Instructions：现成指令池，可直接混用
		- FLAN v2 配方
			- FLAN + P3 + Super-Natural + CoT，20 M 条，领域/任务/长度三重采样，目前 NLP 指令微调首选公共集合
	- （3）基于日常对话数据构建
		- | 来源 | 构建方式 | 特点 | 代表数据集 |
		  |---|---|---|---|
		  | 真实用户查询 | OpenAI API 日志、ShareGPT 多轮对话 | 自然、多样、贴近真实需求 | ShareGPT、OpenAssistant |
		  | 人工扩充 | 雇佣标注员写“开放式生成/问答/头脑风暴”任务 & 答案 | 高质量、可公开 | Dolly、OpenAssistant |
		  | 大模型合成 | 用 ChatGPT/GPT-4 对用户提问生成回复 | 成本低、规模大，需过滤幻觉 | ShareGPT（GPT-4 回复）、WizardLM |
		- 多轮优势
			- 同一 session 的上下文形成连贯指令链，模型学会“追问-澄清-深入”式指令跟随，提升对话场景鲁棒性
		- 质量把控
			- 去 PII、去有害请求
			- 语言分布重采样（避免英语一家独大）
			- 回复长度截断 & 事实核查（减少幻觉）
		- 开源组合配方
			- ShareGPT + OpenAssistant + Dolly → 约 1 M 高质量多轮样本，常被用作“人类风格”指令池，与 NLP 任务模板互补
	- （4）基于合成数据构建
		- 合成指令数据两大利器
			- | 方法 | 核心机制 | 过滤规则 | 产出规模&代表集 |
			  |---|---|---|---|
			  | Self-Instruct | 175 条手写种子 → 迭代 prompt ChatGPT 生成新<指令,输入,输出> | 去重、ROUGE>阈值、过短/过长、无法回答 | Alpaca-52K、Self-Instruct-52K |
			  | Evol-Instruct | 深度演化：5 种 prompt 让指令更复杂（加约束、推理步骤等）<br>广度演化：换领域/少见场景 | ChatGPT 对比相似度、含“sorry”/过短、仅标点 | WizardLM-70K、WizardMath、WizardCoder |
		- 深度演化 prompt 示例
			- “请将下列提示重写为更复杂版本，使得 GPT-4 更难回答，且不得省略原文关键信息：{需重写的原指令}”
		- 广度演化 prompt 示例
			- “创建一条与原文同一领域但更少见的全新提示，合理且人类可理解：{需重写的原指令}”
		- 零成本放大 1→10 万指令，质量接近人工；后续再经过去毒、去幻觉、语言重采样即可投入微调
	- （5）指令数据质量提升方法
		- | 维度 | 做法 | 经验结论 |
		  |---|---|---|
		  | 格式设计 | 零样本（不带示例的指令数据）+少样本（带示例的指令数据）混合；再掺 10–20 % CoT 逐步推理样本 | 同时提升零样本 & 少样本测试，缓解格式敏感 |
		  | 数量扩展 | NLP 任务→7 M 后增益饱和；对话场景 1 k–52 k 高质量即可激活 | Orca 用 GPT-4 逐步回写 5 M 复杂推理样本可继续拔高 |
		  | 重写/筛选 | Evol 深度/广度演化、主题多样性、难度分位数裁剪 | Alpaca-Gasus：从 52 k 筛选 9 k（GPT-4 评分 top）效果不降；YL 难度过滤减少过拟合 |
		- 总结
			- “模板零样本+少量 CoT+GPT-4 重写/筛选” 是目前成本最低、覆盖面最广的指令数据速成方案
			- 数量天花板后，用模型自动演化“更难、更多样”的指令是继续提分的核心路径
	- （6）指令微调的作用
		- | 作用 | 关键结论 | 典型证据 |
		  |---|---|---|
		  | 整体性能改进 | 从77 M 到 540 B 全尺度受益；小模型微调后可反超大模型；架构/目标/方法无关，通用增益 | Flan-T5 11 B 平均 ↑15 %，<br>Flan-PaLM 540 B 平均 ↑9 % |
		  | 任务求解增强&零样本泛化 | 解锁“听懂指令→直接做”能力；缓解重复、补全不解题等问题；英文指令可迁移至多语言 | BLOOMZ-P3 多语言补全 ↑50 % |
		  | 领域专业化 | 医学、法律、电商、金融等继续指令微调即可逼近专业模型 | Med-PaLM 临床问答 ≈医生水平，<br>本草/华佗等中文医学模型 |
		- 成本对比
			- 指令数据量 ≈ 预训练 **万分之一**，算力≅ **1 %** 即可解锁通用任务与领域能力，性价比极高
3. 指令微调的训练策略
	- （1）优化设置
		- 优化器、权重衰减、梯度裁剪、3D并行、ZeRO、混合精度均沿用预训练配置
		- 目标函数：
			- 改用序列到序列损失，仅输出部分计入损失，输入部分被掩码
		- 批次大小/学习率：
			- 大幅减小
			- InstructGPT-175B batch=8、lr=5.03×10⁻⁶恒定
			- Alpaca-7B batch=128、lr 预热至2×10⁻⁵后余弦衰减
		- 多轮对话高效训练：
			- 整段对话一次性前向，通过损失掩码只让助手回复位置参与梯度计算，避免重复前缀计算
	- （2）数据组织策略
		- 平衡采样
			- 把 FLANv2、ShareGPT、GPT4-Alpaca 等先合并，再按固定比例抽签；
			- FLAN v2数据集配比：FLAN 46 %、T0 28 %、NIV2 24 %、CoT 2 %
			- 给每个子集设上限（几千到几万条），防止大数据集垄断
		- 多阶段调度
			- 先拿大规模 NLP 任务指令热启动，再混入日常对话和合成数据，同时回添少量 NLP 样本防遗忘；难度由浅入深，逐步加复杂指令，模型更稳
		- 结合预训练数据与指令微调数据
			- 微调阶段掺 5 % 原始语料（OPT-IML 做法），可缓解生成能力下降
			- GLM-130B 把 5 % 指令任务提前混合进预训练
			- MiniCPM 则在两阶段之间加“退火期”：高质量语料与指令数据 1:1 混合继续训练，效果优于单纯先预训后指令
4. 参数高效的模型微调
	- （1）参数高效微调（Parameter-efficient Fine-tuning）
		- 也称为轻量化微调 （Lightweight Fine-tuning）旨在减少需要训练的模型参数量，同时保证微调后的模型性能能够与全量微调的表现相媲美
	- （2）低秩适配微调方法（Low-Rank Adaptation, LoRA）
		- LoRA 原理
			- 核心假设
				- 下游任务对预训练权重 $\mathbf W_0\in\mathbb R^{H\times H}$ 的更新 $\Delta\mathbf W$ 存在低秩结构，用 $\Delta\mathbf W=\mathbf A\mathbf B^\top$ 近似，$\mathbf A\in\mathbb R^{H\times R},\,\mathbf B\in\mathbb R^{H\times R},\,R\ll H$  
			- 前向改写：
				- $\mathbf h=\mathbf W_0\mathbf x+\mathbf A\mathbf B^\top\mathbf x$，微调阶段仅 $\mathbf A,\mathbf B$ 可训，$\mathbf W_0$ 冻结
			- 推理合并：
				- 微调结束把 $\mathbf W=\mathbf W_0+\mathbf A\mathbf B^\top$一次性加回，无额外延迟，显存和计算与原始模型完全一致，推理零额外开销  
			- 训练参数量：
				- 从 $H^2$ 降到 $2HR$，通常 $R=4$~$64$，显存下降 90 %+
		- LoRA 显存
			- 全量微调：
				- $16P$ 字节（参数+梯度+Adam 状态全部参与）
			- LoRA 微调：
				- $2P$（模型原参数） + $16P_{lora}$ 字节，
				- $P_{lora} = 4 × 2LRH$（仅 $Q/K/V/O$ 四个矩阵加低秩R旁路）
			- LLaMA-7B 例
				- $L=32, H=4096, R=8 → P_{lora}≈8.4×10⁶$
				- 显存从 108 GB 降到 14 GB，近似“2P”规律：轻量化后仅需保存原模型 FP16 副本与小秩矩阵
		- LoRA 变种
			- AdaLoRA：
				- 训练中以损失敏感度动态调整每矩阵的秩：重要矩阵给高秩，次要矩阵给低秩，参数总量可控，效果优于固定秩
			- QLoRA：
				- 先把原权重量化到 4 bit，低秩旁路仍用 16 bit 训练，显存从 $2P$ 再降到 $0.5P$
				- 单卡 $A6000$（48 GB）即可微调 65 B 模型，性能接近 16 bit 微调
	- （3）其他高效微调方法
		- 在预训练语言模型中被广泛使用，但是在大语言中的应用相对较少
		- 适配器微调
			- 在 Transformer 每层SA和FFN后插入“瓶颈”子网络（适配器）：先降维 $R≪H$，激活后再升维回原维度  
			- 前向：$h ← h + σ(h·W_d)·W_u$，仅训练 $W_d∈ℝ^{H×R}, W_u∈ℝ^{R×H}$；原参数冻结
			- 参数量：$2LRH$（L 层数），显存与 LoRA 同级；串行结构带来 < 5 % 推理延迟
		- 前缀微调（Prefix Tuning）
			- 在每层多头注意力键、值前拼接可训练前缀矩阵 $\mathbf P_K,\mathbf P_V\in\mathbb R^{L\times H}$，注意力变为$\mathrm{head}=\mathrm{Attention}(\mathbf X\mathbf W_Q,\, [\mathbf P_K;\mathbf X\mathbf W_K],\, [\mathbf P_V;\mathbf X\mathbf W_V])$ 
			- 仅 $\mathbf P_K,\mathbf P_V$ 参与梯度更新，原参数冻结；$L$ 为前缀长度，通常 10~100，显存增量可忽略
			- 重参数化技巧：
				- 通过 MLP 将较小的矩阵 $\mathbf P_\mathrm{small}$ 映射到前缀参数矩阵 $\mathbf P$
				- 实际优化小矩阵 $\mathbf P_\mathrm{small}$，稳定训练；推理时丢弃 MLP，只保留最终 $\mathbf P$
		- 提示微调
			- 仅在输入嵌入层插入可训练“软提示”向量 $\mathbf P\in\mathbb R^{L\times H}$，与文本嵌入拼接后一次性送入模型，后续网络全部冻结  
			- 自由形式（P-tuning）
				- 使用自由形式来组合输入文本和提示向量，用 LSTM 学习 $\mathbf P$ 的排列
			- 前缀形式（Prompt Tuning）
				- 直接把 $\mathbf P$ 拼在句首；参数量最少 $L\times H$，效果高度依赖底座模型规模
5. 代码实践与分析
	- （1）指令微调的代码实践
		- 与预训练共用 DeepSpeed + ZeRO-3 + BF16 + 激活重计算，唯一区别：
			- 数据集换成 `SFTDataset`（每条样本 = 指令 + 输入 + 输出，仅输出计算 loss）
			- 数据_collator 改用 `DataCollatorForSupervisedDataset`，自动掩码输入段
		- 多卡启动：`torchrun --nproc_per_node=8 train_sft.py --per_device_batch_size 8 --max_seq_len 512 --num_train_epochs 3 --deepspeed ds_z3_config.json`
		- 成本实测（LLaMA，Alpaca-52k，512 长度，3 epoch，A800 80G）：
			- $7B$：2 卡，batch=16，  ≈ 2.0 h
			- $13B$：4 卡，batch=8，  ≈ 3.1 h
			- $30B$：8 卡，batch=4，  ≈ 6.1 h
			- $65B$：16 卡，batch=2，≈ 11.2 h
		- 经验：全量微调至少留 16P 字节显存；若资源不足，LoRA/QLoRA 可把显存压到 2P/0.5P
	- （2）指令微调实验结论（LLaMA-2 7B vs 13B，AlpacaFarm + MMLU + BBH）
		- 任务对口才有效
			- 使用与下游任务格式更接近的指令能够带来更大提升
			- FLAN v2 擅长 NLP 任务（MMLU ↑），却在日常对话败给 ShareGPT；后者真实用户 query 多，对话胜率领先 10+ 点，但 NLP 基准略弱
		- 复杂化 + 多样化
			- 提高指令复杂性和多样性能够促进模型性能的提升
			- WizardLM-70K（Evol 复杂化）与主题多样化 70K 把 Alpaca-52K 的对话胜率从 46.5 → 52-58 %，逼近 ShareGPT
			- 同时对 MMLU/BBH 也有小幅增益，证明“更难、更多主题”的指令可弥补数据真实度差距
		- 参数规模仍是硬通货
			- 更大的参数规模有助于提升模型的指令遵循能力
			- 同配方下 13B 全面压制 7B：
				- ShareGPT 在 MMLU 上 49.7 → 56.8（+7.1），BBH 同步提升；
				- 复杂/多样增强后对话胜率再涨 4-6 点，大底座更能吸收高难指令
	- （3）LoRA 代码实践与分析
		- 实战调用：
			- HuggingFace 的 PEFT 参数高效微调代码库，支持多种的高效微调方法，包括 LoRA和AdaLoRA、Prefix Tuning、P-Tuning和Prompt Tuning
			- PEFT库代码流程：
			  `LoraConfig(task_type="CAUSAL_LM", r=16, lora_dropout=0.05)` → `get_peft_model()` → 训练 → `merge_and_unload()` 合并权重  
		- 资源实测（LLaMA，Alpaca-52k，3 epoch，512 长度，r=16）
			-  $7B$：1 卡，batch=16，≈ 2.3 h
			- $13B$：1 卡，batch=8， ≈ 3.8 h
			- $30B$：1 卡，batch=1， ≈ 10.2 h
			- $65B$：2 卡，batch=1， ≈ 26.0 h
		- 显存底线：
			- 大约 2P 即可开练，较全量16P节省 87%；大模型本身 2P 已接近80G，batch 只能为 1，时间成本反而升高，可考虑 QLoRA 再砍一半显存
		- 底层的代码实现
			- ```python
			  import torch
			  import torch.nn as nn
			  import torch.nn.functional as F
			  # 继承PyTorch的线性变换类
			  class LoRALinear(nn.Linear):
				  def __init__(self,in_features,out_features,config,bias=True):
					  super().__init__(in_features,out_features,bias=bias)
					  # 从配置中获取LoRA的秩，这决定了低秩矩阵A和B的大小
					  self.r = config.lora_r
					  
					  # 初始化A，将输入映射到低秩空间r
					  self.A = nn.Linear(in_features,self.r,bias=False)
					  #初始化B，将低秩空间映射回原始输出空间
					  self.B = nn.Linear(self.r,out_features,bias=False)
					  
					  # 初始化一个丢弃层，用于在输入传递给A之前进行正则化
					  self.dropout = nn.Dropout(p=config.lora_dropout)
					  
					  # 使用标准差为0.02的正态分布初始化A的权重
					  self.A.weight.data.normal_(std=0.02)
					  # B的权重初始化为零
					  self.B.weight.data.zero_()
					  # 
				  
				  def forward(self, input):
					  # 原始权重self.weight, self.bias冻结，只算一次乘法，不参与更新
					  linear_output = F.linear(input, self.weight, self.bias)
					  # LoRA模块对应输出
					  lora_output = self.B(self.A(self.dropout(input)))
					  # 将标准线性输出与缩放后的LoRA输出相加，得到最终输出
					  return linear_output + lora_output
			  ```
		- 说明：B 初始化为 0 即可保证训练开始时 ΔW=0；把 A 设成小的非零随机数，能让梯度正常回传，于是训练才能启动
