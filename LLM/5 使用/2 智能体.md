1. 基于大语言模型的规划
	- （1）整体框架
		- 核心思想：
			- 把复杂任务拆成可顺序求解的子任务，用“规划器-执行器-环境”三元组显式解耦规划与执行
		- 三大组件
			- 任务规划器（LLM）：
				- 生成自然语言或代码形式的动作序列；可带存储模块支持长期任务
			- 规划执行器：
				- 负责落地动作，可由 LLM、机器人或其他实体担任
			- 环境：
				- 提供动作执行场所（Web、Minecraft、物理世界等）并返回反馈
		- 流程：
			- 规划器一次性或迭代地输出方案 → 执行器在环境中运行 → 反馈回流 → 规划器迭代优化
		- 与 RL 区别：
			- 规划与执行分离，非耦合在单一智能体；更强调显式方案生成与可解释性
	- （2）方案生成：一次性 vs 迭代式
		- 一次性生成
			- 做法：LLM 在单轮提示里直接输出完整动作序列（自然语言或代码）
			- 优点：零交互、延迟低；适合短、确定性任务
			- 风险：中间一步错则全局失败；无反馈修正机会
		- 迭代式生成（ReAct / Reflexion）
			- 做法：每步仅生成“下一步动作+理由”，执行后获得环境反馈，再循环
			- 优势：可实时纠错、回退（backtrack），对长程或动态任务更鲁棒
			- 关键技巧：用“思考→动作→观察”模板提示，配合奖励或自我批判触发回退
			- 选型建议：强逻辑/数值任务优先代码表达；开放式问答优先自然语言
	- （3）反馈获取：外部 vs 内部
		- 外部反馈
			- 工具：代码解释器即时返回运行结果/报错，LLM 据此修正后续代码
			- 人类：具身场景中人实时回答状态查询，补充机器人感知盲区
			- 虚拟环境：游戏或模拟器返回动作执行后的状态变化，供下一步规划
		- 内部反馈（自反）
			- 自检：用提示“Is the current action correct?”让 LLM 直接评判历史动作及其反馈的合理性
			- Reflexion工作：对已执行的方案及其反馈（成功/失败）通过任务规划器给出自然语言“反思”，再和已执行方案及其反馈一起给任务规划器规划新一轮方案，直至得到成功的反馈
			- 效果：将稀疏外部信号转化为密集语义信号，减少反复试错，提升长程任务成功率
2. 基于大语言模型的智能体
	- （1）智能体概述
		- 智能体（Agent）是一个具备环境感知、决策制定及动作执行能力的自主算法 系统
		- 从规则到 LLM：智能体演进简史
			- 规则时代：依赖专家手工写 if-then，封闭域有效，遇新场景立即失效
			- 模型时代：深度学习提取特征+强化学习试错，策略可泛化，但仍需大量交互样本
			- LLM 时代：用语言模型同时充当“知识库+推理器+规划器”，零样本/少样本即可开放域决策
			- 关键跃迁：提示工程替代特征工程，自然语言成为统一动作接口，大幅降低部署门槛
	- （2）大语言模型智能体的构建
		- 记忆组件 Memory
			- 短期记忆：
				- 即上下文窗口，存当前回合的观察/对话，容量有限且随会话结束衰减
			- 长期记忆：
				- 外存（文本、向量库、结构化 DB），存用户画像、事实、经验；支持读写检索更新与遗忘机制，容量近乎无限
			- 调用方式：
				- 用提示模板“读取→写入→压缩→遗忘”，保证相关历史随取随用
		- 规划组件 Planning
			- 任务分解：
				- 将用户指令拆成可执行子目标，可一次性输出完整方案或 ReAct 式逐步生成
			- 多方案评估：
				- 自生成 k 个候选计划，用LLM自评或投票选最优；支持回溯与重规划
			- 输入源：
				- 同步读取长短期记忆 + 环境反馈，保证规划随情境动态更新
		- 执行组件 Execution
			- 内置执行：
				- 直接用 LLM 生成自然语言或代码完成动作（如调用 API、发消息）
			- 外置工具：
				- 集成代码解释器、机械臂、浏览器等，突破纯文本限制
			- 闭环：
				- 执行方案→环境反馈→短期记忆→规划组件，形成“感知-决策-行动”循环
		- 标准工作流程（单轮视角）
			- 步骤 1：状态理解
				- 检索长期记忆 + 当前输入 → 形成对环境的完整认知
			- 步骤 2：规划
				- 基于记忆生成 k 个候选动作 → 自评或打分选最优
			- 步骤 3：执行
				- 调用内置 LLM 或外部工具完成动作
			- 步骤 4：反馈
				- 环境返回原始信号（成功/失败/异常/观察）
			- 步骤 5：记忆更新
				- 过滤无关信息→反馈写入短期记忆 → 短期记忆同步 flush 到长期记忆
			- 步骤 1~5 循环：
				- 用更新后的记忆重新进入步骤 1，直到任务目标达成或达到最大轮次
	- （3）多智能体系统构建要点
		- 构建步骤
			- 明确总目标：
				- 任务求解 or 环境仿真
			- 实例化智能体：
				- 同源或异源 LLM，赋予角色（规划 / 评判 / 知识源）
			- 定义交互：
				- 协作、竞争、协商；用协议固定通信格式、频率、时序
			- 保障属性：
				- 可扩展（动态增减节点）
				- 可维护（日志 / 调试接口）
				- 安全（加密 + 鉴权）
				- 异构兼容（不同模型 / 工具链）
		- 通信三要素
			- 协议：
				- 定义如何进行信息交换和共享；自然语言、JSON、代码片段皆可；需统一 schema 防止歧义
			- 拓扑：
				- 定义智能体之间的连接关系；全连接、星型、链式或隐式通过共享内存/blackboard；拓扑会影响带宽与一致性
			- 内容：
				- 状态、指令、任务目标、置信度；支持增量更新与压缩以减少 token 开销
		- 协同三模式
			- 协作：
				- 共享资源与信息，用投票 / 共识 / 任务分配算法提升整体 reward
			- 竞争：
				- 资源受限时引入博弈论和竞价机制，系统层面可达纳什最优
			- 协商：
				- 出现目标或资源冲突则多轮让步-交换，直至 Pareto 改进收敛
		- 运行要求：
			- 实时性（低延迟通信）
			- 可靠性（丢包重传 / 校验）
			- 安全性（保护通讯内容；端到端加密 & 权限隔离）
	- （4）大语言模型智能体的典型应用
		- WebGPT（单智能体）
			- 环境：文本式网页浏览器
			- 动作：搜索、点击、收藏、引用相关网页信息，对搜索结果进行分析整合
			- 结果：端到端问答，引用可溯，人类评估准确率≈真人水平
		- MetaGPT（多智能体）
			- 角色：产品经理→架构→项目经理→开发→测试，全流程 SOP 
			- 协作：标准化输出（需求文档/UML/代码）+ 消息队列共享
			- 成效：分钟级生成可编译项目；缺陷：部分代码无法直接运行需二次调试
		- 《西部世界》沙盒（生成式智能体）
			- 设定：25 个代理，每人一段自然语言人设（配置文件），沙盒小镇 2 天实时模拟
			- 行为：自发约会、办画展、传播谣言；记忆链保留 5k+ 事件
			- 意义：验证“纯语言驱动”社会动力学，为推荐、搜索提供高保真用户仿真数据
		- 开发捷径：
			- AgentScope、RecAgent 等开源框架已封装记忆、通信、角色模板，可直接拼装业务 MAS（Multi-Agent System）
	- （5）待解决的关键技术问题
		- 计算资源黑洞
			- 单轮即一次完整LLM调用，多agent并行时调用量线性爆炸；
			- 亟需模型压缩、量化、MoE路由及调度池化
		- 复杂工具适配
			- 工具接口与LLM语义不对齐→调用失败率高；
			- 需统一工具描述协议（JSON Schema/Function Calling）+自动文档生成，实现“零样本”新工具接入
		- 大规模agent通信
			- 节点>100时全连接O(n²)消息风暴；
			- 研究拓扑稀疏化、分层共识、黑板/消息总线机制，保证实时性与扩展性
		- 模型本体适配
			- 长窗口幻觉、角色一致性漂移（身份与行为一致性逐渐弱化）；
			- 通过指令增强、记忆分段检索、角色embedding固化，对基座模型继续轻量微调（LoRA/adapter）以强化“agent能力包”
		- 真实世界落地
			- 虚拟→物理的Sim-to-Real差距：感知误差、执行误差、安全合规；
			- 需引入数字孪生预演、不确定性量化、人类价值对齐、故障熔断机制
