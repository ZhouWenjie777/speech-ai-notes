1. 解码策略
	- （1）解码
		- 模型针对输入内容（提升文本）逐个单词生成输出内容的文本
		- 本质上是一个概率采样过程，需要合适的解码策略来生成合适的内容
	- （2）自回归解码基础
		- 流程：
			- 模型逐步输出概率分布 $P(a|context)$ → 解码策略选词 → 拼回上下文再循环，直到 `EOS` 或最大长度  
		- 贪心搜索：
			- 每步取 $arg max$，速度快但易陷入局部最优，生成重复、不自然文本  
		- 概率采样：
			- 按 $P$ 随机抽签，引入随机性，提升多样性；缺点是可能采到低概率词，连贯性下降
	- （3）贪心搜索三大补丁
		- 束搜索（Beam Search）：
			- 保留 top-n 条最高概率前缀，每步基于联合概率排序；n=3~6 平衡质量与速度；n 过大反而因长度偏差掉分  
		- 长度惩罚：
			- 联合概率除以长度^α（α≈0.6–0.7），抵消短句概率天然偏高，鼓励输出完整长句
		- 重复惩罚：
			- 𝑛-元惩罚：使用 𝑛-元惩罚（𝑛-gram Penalty）来强制避免生成重复的连续 𝑛 个词元
			- 出现惩罚（Presence Penalty）：对已生成 token 的之后 `logits` 减 α，降低该词元之后生成的概率
			- 频率惩罚（Frequency Penalty）：减去出现次数乘以惩罚项α，出现次数越多对联合概率惩罚越大，适用于所有解码策略
	- （4）随机采样的改进策略
		- 温度采样：
			- 在 softmax 里显式加温度系数，$P(u_t|u_{<t})=\frac{\exp(z_t/T)}{\sum_{v\in V}\exp(z_v/T)}$；
			- $z_t$ 是`logits`，T 由人工设定（常用 0.1~2.0），T→0 趋贪心，T→1 标准随机采样，T→∞ 趋均匀；T<1 峰更尖，高概率词占优，低概率词被压缩
		- Top-k：
			- 只保留概率最高的 k 个词再重归一化；
			- k 小 → 保守，k 大 → 多样；无法随分布动态调整  
		- Top-p（核采样）：
			- 按概率从高到低累加，取最小集合使累加≥p；p=0.8 表示“核心 80 %概率质量”，集合大小随上下文自动伸缩，兼顾保真与多样  
		- 对比解码：
			- 用大模型 `logits` 减去小模型 `logits`，再 softmax；放大“大模型显著优于小模型”的 token 概率，抑制共现高频但信息量低的词，提升事实性与关键信息命中率
	- （5）实战解码配置一览
		- | 模型/场景 | 策略 | 关键参数 |
		  |---|---|---|
		  | T5 翻译/摘要 | 束搜索 | beam=4, length_penalty=0.6 |
		  | GPT-3 通用 | 束搜索 | beam=4, length_penalty=0.6 |
		  | Alpaca 开放生成 | top-k + top-p + 温度 | k=50, p=0.9, T=0.7 |
		  | LLaMA 问答 | 贪心搜索 | — |
		  | LLaMA 代码(pass@1) | 温度采样 | T=0.1 |
		  | LLaMA 代码(pass@100) | 温度采样 | T=0.8 |
		  | OpenAI API | 贪心/束/温度/top-p | temperature=0→贪心, best_of→束, top_p→核采样, presence_penalty/frequency_penalty∈\[0,2\] |
		- 效果示例（YuLan-7B，同一 prompt）
			- 贪心：信息全但稍冗长  
			- 贪心+频率惩罚 α=1：去掉重复短语，更简洁  
			- top-p=0.95：温度式表达，给出区间值，多样性↑
2. 解码加速算法
	- （1）解码效率分析
		- 解码两阶段与 KV 缓存
			- 全量阶段（prefill）：
				- 对输入 u 一次性前向，算出首 token 的 Q、K、V 并缓存 K_past、V_past  
			- 增量阶段（decode）：
				- 每步仅把新 token $u'$ 过一遍网络，与 **缓存的 K_past、V_past** 做注意力；得到输出后把新 $K'$、$V'$ 拼到缓存，继续下一循环
			- KV 缓存结果：
				- 二次复杂度从 O(L²) 降到 O(L)（L 为已生成长度），显存换时间
		- 解码效率定量指标
			- GPU 计算强度上限：$I_{\max}=\frac{\text{GPU 算力 (FLOP/s)}}{\text{GPU 带宽 (byte/s)}}$；
			- A100 例：$I_{\max}=\frac{3.12\times10^{14}}{2039\times10^9}\approx142.5\ \text{FLOP/byte}$  
			- 模型计算强度：$I=\frac{\text{运算量 (FLOP)}}{\text{访存量 (byte)}}$  
			- 瓶颈判断：
				- $I<I_{\max}$ → 带宽瓶颈（KV 缓存、embed 搬运占主导）
				- $I>I_{\max}$ → 计算瓶颈（大矩阵乘 FLOP 过剩）  
			- 解码阶段因序列平方级 Attention 读写，$I$ 随长度下降，大模型长文本生成常陷**带宽瓶颈**；提升手段：KV 缓存、量化、tensor 并行、高带宽内存
		- 计算强度实例（LLaMA-7B, A100 80 GB）
			- 全量解码：
				- 线性变换 I≈2730，多头注意力 I≈114，均接近 A100 上限 142.5 → **计算瓶颈**
			- 增量解码：
				- 同一模块 I≤8，远小于 142.5 → **带宽/内存墙瓶颈**（俗称 Memory-bound）  
			- 结论：  
				- 全量阶段受限于 **FLOPS**；增量阶段受限于 **显存读写**  
				- 优化重心应放在 **KV-cache 压缩、量化、张量并行、高带宽内存** 等内存侧手段，而非继续堆算力
	- （2）系统级“内存墙”破解方案
		- FlashAttention：
			- 核心思路：尽可能减少注意力计算中对于中间结果的保存，如$𝑸𝑲^T$和softmax后的注意力矩阵，大幅减少显存读写量
			- 分块+算子融合，把中间结果注意力矩阵留在缓存 SRAM，最终结果才写回显存；
			- LLaMA-2-7B L=2048 Batch =8 场景注意力耗时降至 1/10  
		- PagedAttention：
			- 有效针对键值缓存拼接和注意力计算的优化操作，有效降低访存量
			- 预分页管理 KV-cache，拼接不再 malloc分配/拷贝，减少碎片与读写；
			- 使用算子融合方法并行计算当前词元查询与多页KV，提高 decode 并行度  
		- 连续批处理（Continuous Batching）：  
			- 把请求拆成“全量阶段 + 单步增量阶段”，启发式混排，一步内塞更多样本 → 有效 batch 变大，计算强度↑
			- vLLM代码库 实测 7B 模型吞吐提升 2–4 倍，首 token 延迟下降 50 % 
		- 动态分割（DeepSpeed-MII代码库）：
			- 进一步把全量 decode 切分为多个子操作，与增量 decode 同批执行，粒度更细，进而获得更大的 batch 和更高的解码吞吐量
	- （3）解码策略优化
		- 推测解码（Speculative Decoding）
			- 小模型（如n元统计模型或者小型预训练模型）快速自回归生成 k 个 token → 大模型一次并行验证；错位即修正，正确即接受；k≈3~5，实测 1.5–2× 加速，质量无损
			- 验证用 1 次 k token前向 替换 k 次 1 token前向，利用 GPU 高并行度换时间
		- 级联解码（Cascade Inference）
			- 按效率从高到低排队小→中→大模型；轻量模型输出满意即提前返回，减少平均延迟；用二分类器判断“是否满意” ，可设定分类器的判断阈值
		- 非/半自回归（Non-autoregressive Decoding）
			- Medusa 在 Vicuna 上加 2 个额外预测头，一次性并行生成 3 个 token → 原模型验证；可嵌到 推测解码 里当“草稿模型”
		- 早退机制（Early Exiting）
			- 每层Transformer输出接预测头得到词表概率分布，分布熵<阈值（即某一个词的概率显著较高）即提前出词；
			- 或“混合深度方法”用路由网络跳过部分层；每一层的输入通过路由网络计算得分，得分<阈值即跳过该层；层计算量↓20–40 %，质量几乎不掉
	- （4）解码代码实践
		- llama.cpp：
			- 纯 C/C++ 实现，跨 CPU/Apple/NVIDIA/AMD，支持 1.5–8 bit 量化，边缘设备友好  
		- vLLM：
			- 分页 KV-cache + PagedAttention，连续批处理，吞吐量×2–4；API 兼容 OpenAI，一行命令拉起服务  
		- DeepSpeed-MII：
			- 动态分割全量/增量阶段，同一批次混排，进一步增大有效 batch；支持张量并行，多卡推理  
		- FlexFlow：
			- 设计树状注意力机制，将多组推测草稿拼接到一条序列中，通过修改注意力掩码一次验证多组草稿，推测解码并行度↑，小模型 draft 次数利用率↑  
		- 快速体验（vLLM 示例）
		  ```bash
			  # 启动服务
			  python -m vllm.entrypoints.api_server \
				  --model meta-llama/Llama-2-7b-chat-hf \
				  --dtype bfloat16 --tensor-parallel-size 1 --port 8000
			  
			  # 调用（与 OpenAI API 格式一致）
			  curl http://localhost:8000/v1/completions \
				  -H "Content-Type: application/json" \
				  -d '{
						"prompt":"[INST] Tell me a joke [/INST]",
						"max_tokens":50,
						"temperature":0.7
					}'
		  ```
3. 低资源部署策略
	- （1）量化基础知识
		- 映射公式：
			- 浮点到定点量化 $X_q = \text{round}(X/S) + Z$
			- 定点到浮点反量化 $\hat X = S(X_q - Z)$
			- 误差 $e = X - \hat X$，需离线校准或在线统计确定 S、Z
		- 均匀 vs 非均匀
			- 均匀：量化等级等间距，公式即上式，硬件友好，主流 INT8/INT4 皆属此类
			- 非均匀：等级按数据分布（如累积分布）变间距，需查表或特殊电路，精度高但实现复杂，仅用于极端低比特（≤2 bit）
		- 对称 vs 非对称：
			- 对称（Z=0）：零点映射到 0，适合数值近似零中心分布 → INT8 范围 \[−128, 127\] 
			- 非对称（Z≠0）：可偏移，适合全正/负分布 → \[0, 255\]
		- 量化粒度：
			- 按张量（per-tensor）：整网共用一对 (S,Z)，实现简单，误差最大
			- 按通道（per-channel）：每列/通道独立参数，精度↑，显存额外占比 <1 % 
			- 按组（per-group）：通道再细分，精度再↑，计算开销随之增大；主流选择 per-channel 兼顾准确与速度
		- 量化示例与代码
			- 非对称
				- 给定 $x∈[1.2,13.6]$，目标 $INT8∈[-128,127]$
				- 解方程：$S(127-Z)=13.6；S(-128-Z)=1.2$；$S≈0.0486,Z≈-152$
			- 对称
				- Z=0，覆盖$[-13.6,13.6]→S≈0.1067$；区间浪费+步长变大，误差大于非对称
	- （2）大模型训练后量化方法
		- 大模型量化两条路线
			- QAT（量化感知训练）：
				- 微调阶段插入伪量化节点，前向用低精度、反向用浮点，同步更新权重；精度高，但需全量训练，算力×1–2，百亿模型极少采用  
			- PTQ（训练后量化）：
				- 权重校准 → 离线量化 → 零重训；小时级完成，成为 LLM 部署主流；难点在激活异常值（outliers）使低比特（≤4 bit）误差爆炸
			- PTQ 核心挑战
				- 权重分布近似高斯，易量化；激活值存在极大 token-level outliers，传统 MinMax/对称量化会把大部分有效区间压缩，导致精度雪崩  
				- 解决思路：逐通道、动态离线校准、异常值隔离、混合精度（权重 INT4 + 激活 INT8）、KV-cache 量化等，后续展开
		- 权重量化代表方法
			- GPTQ：
				- 逐层 + 列分组（如 128 列一组），量化一列后立即用 **Hessian 逆** 更新同组剩余列，补偿误差；
				- 延迟批更新 + Cholesky 分解，3–4 bit 下 7 B 模型 Perplexity 几乎不掉  
			- AWQ：
				- 仅 0.1 %–1 % 权重对激活大值维度“关键”，给这些通道加 **激活感知缩放因子 s**；目标改写为 $$\min_W\bigl\|\bigl(\text{diag}(s)^{-1}X\bigr)\cdot Q\bigl(W\cdot\text{diag}(s)\bigr)-XW\bigr\|_2^2$$
				- 其中 $Q$ 为量化函数，量化前放大关键通道，量化后缩回，使重要权重落在量化网格中心，3 bit 精度优于 GPTQ，且无需 Hessian 逆，运行时间↓10×
		- 权重+激活值联合量化方案
			- 细粒度量化
				- ZeroQuant对激活值**逐 token**动态校准；
				- 权重**按组**（128 列）量化，减少粗粒度误差  
			- 混合精度分解（`LLM.int8()` 思路）
				- 异常值维度（≈0.1 %）直接 FP16 计算：$\mathbf O_{\text{FP16}} = \mathbf X_{\text{FP16}}\mathbf W_{\text{FP16}}$
				- 正常维度先逐行/逐列缩放→INT8 矩阵乘→32位整数累加→反量化为FP16
				- 公式链：$$
			  \begin{aligned}
			  \mathbf X_{\text{INT8}} &= \lfloor 127\cdot\text{diag}(s_x)^{-1}\mathbf X_{\text{FP16}}\rceil \\[2pt]
			  \mathbf W_{\text{INT8}} &= \lfloor 127\cdot\mathbf W_{\text{FP16}} \text{diag}(s_w)^{-1}\rceil \\[2pt]
			  \mathbf O_{\text{INT32}} &= \mathbf X_{\text{INT8}}\mathbf W_{\text{INT8}} \\[2pt]
			  \mathbf O_{\text{FP16}} &= \frac{s^T_x s_w}{127^2}\odot\mathbf O_{\text{INT32}}
			  \end{aligned}
				  $$
				- 其中 $𝒔_x$ 和 $𝒔_w$ 分别表示输入的激活值和权重中每一行/每一列中绝对值的最大值
				- $diag$ 表示将向量中元素排列在对角线上，其他位置为零，形成对角矩阵
				- 两部分结果相加得最终 FP16 输出；峰值显存降 50 %，推理速度↑ 1.2–1.5×，精度几乎无损
			- 量化难度平衡（SmoothQuant）
				- 核心思想：
					- 把“难量化”的激活异常值 **数学等价地** 搬到权重侧
					- 线性层变换：$\mathbf Y = \bigl(\mathbf X\cdot\mathrm{diag}(\mathbf s)^{-1}\bigr)\cdot\bigl(\mathrm{diag}(\mathbf s)\cdot\mathbf W\bigr)$ 
					- 缩放因子：$s_j = \max(|x_j|)^\alpha\big/\max(|w_j|)^{1-\alpha}$（α=0.5 常用）  
				- 效果：激活值维度异常值被“平滑”，权重承担部分幅度 → 二者均可 INT8 量化，精度几乎不掉；在 175 B 模型上实现 1.5× 吞吐提升，显存↓ 30 %  
				- 实现：仅需离线校准一次s，推理时fuse进线性层权重，零额外延迟
		- 其他量化路线
			- 高效微调增强：
				- QLoRA = 4-bit 量化主模型 + 16-bit LoRA 适配器补偿；低秩参数补偿量化误差，7B/65B 在下游任务上达到 16-bit 全量微调 99 % 性能，显存再降 50 %  
			- 量化感知训练（QAT）大模型：
				- 用教师模型输出分布蒸馏，同时更新量化权重；LLaMA 实验显示 权重 + KV-cache 4 bit 量化可达可用水平，激活 4 bit 仍掉分，需要更细粒度混合精度或异常值隔离方案
		- 基于开源代码库的YuLan量化实战
			- `bitsandbytes`库：
				- HF 一行 `load_in_8bit=True/4bit=True` 即完成 INT8/INT4 权重量化；
				- 支持 `NF4/FP4` 混合精度，显存 13 B → 6 GB（4 bit），解码速度↑1.3×，可直接接 QLoRA 微调  
			- `GPTQ-for-LLaMA`库：
				- 专供 GPTQ 4 bit，单卡 RTX 3090 24 G 可跑 7 B 推理；
				- 官方提供与 `bitsandbytes` 的 Perplexity(PPL) vs 显存对比表，
				- 4 bit 下二者 PPL 几乎重叠，`GPTQ` 显存再省 10 %  
			- 选型建议：快速部署/微调首选 `bitsandbytes`；极致压缩或旧 GPU 选 GPTQ；两种库都与 HF `device_map="auto"` 无缝兼容，零代码侵入
	- （3）经验性分析与相关结论
		- INT8 权重量化
			- 几乎不掉性能，显存减半，可闭眼启用  
		- INT4/3 权重量化
			- 需 AWQ/GPTQ 等激活感知或逐列补偿；同系列“4-bit 60 GB”模型往往优于“8-bit 30 GB”，上下文学习、推理、指令跟随能力损失极小  
		- 激活值量化
			- 参数规模超过某个阈值（如6.7 B）后出现异常值，必须混合精度/逐 token/难度迁移；INT8 可行，INT4 仍待研究  
		- 超低比特补偿
			- 2-bit 权重 + 16-bit LoRA/QLoRA 微调，可把 65 B 量化模型拉回 13 B 全16-bit精度水平；
			- QLoRA 新增 NF4 类型与分页优化器，4-bit 微调≈全量 16-bit 效果  
		- 实战建议：显存受限先上 4-bit 权重量化；若任务掉分，再叠加 LoRA/QLoRA 微调恢复，成本最低
4. 其他模型压缩方法
	- 模型蒸馏和模型剪枝都是通过精简模型的结构，进而减少参数的数量
	- 模型蒸馏
		- 目标：把大模型（教师）的知识“压”进小模型（学生），参数↓，推理↑
		- 知识来源：
			- 基于反馈：用教师最终 softmax 输出（logits）当“软标签”，温度 T>1 放大细微差异，与学生 logits 做 KL 散度
			- 基于特征：强迫学生中间隐藏态与教师对齐，额外 MSE 或 cosine 损失 
		- 损失组合：$\mathcal L=\mathcal L_{\text{task}}+\lambda\,\mathcal L_{\text{KL}}+\mu\,\mathcal L_{\text{feat}}$，联合微调
		- 传统知识蒸馏方法
			- 基于反馈（logits）
				- $$\mathcal L_{\text{logit}} = \tau^2\cdot\mathrm{KL}\!\left(\mathrm{softmax}(\mathbf z_t/\tau)\parallel\mathrm{softmax}(\mathbf z_s/\tau)\right)$$
				- 温度 $τ>1$ 放大软分布差异；$τ²$ 因子抵消梯度量级随 $τ$ 的缩放
			- 基于特征（hidden）
				- $$\mathcal L_{\text{feat}} = \bigl\|\phi(\mathbf h_t)-\phi(\mathbf h_s)\bigr\|_2^2$$
				- $φ$ 通常 1×1 线性投影或插值，用于对齐通道/长度差异，处理输出形状不匹配的情况
				- 可选多层加权平均
			- 联合训练
				- $$\mathcal L = \mathcal L_{\text{task}} + \lambda\mathcal L_{\text{logit}} + \mu\mathcal L_{\text{feat}}$$
				- $λ, μ$ 经验 0.3~0.7；教师模型冻结，仅学生参数更新
		- 大模型蒸馏方案
			- 白盒 MINI LLM：
				- 可访问教师权重，用逐层 MSE + 隐藏态协方差对齐，13 B → 7 B 保持 95 % 下游任务性能  
			- 黑盒 CoT 蒸馏：
				- 仅调用教师 API，把“问题 + 选项”喂给大模型，让其输出完整思维链；学生模型联合优化 $$\mathcal L=\mathcal L_{\text{label}}+\lambda\mathcal L_{\text{cot}}$$
				- 标签损失 + 思维链文本生成损失同步回传；770 M T5 在 ANLI 仅用 80 % 样本即达标准微调全部样本的效果  
			- 关键能力重点：上下文学习、CoT 推理、指令跟随均可按此范式蒸馏； logits 过大时可温度缩放或分段采样，降低 API 调用成本
	- 模型剪枝
		- 目的：在尽可能不损失模型性能的情况下，努力消减参数数量，有效降低显存需求以及算力开销
		- 传统剪枝双模式
			- 结构化剪枝：
				- 整通道/头/维度删除，L2 范数度量重要性；剪完矩阵变小，乘加次数真减少，**硬件直接加速**
				- PyTorch 一行：`torch.nn.utils.prune.ln_structured(module, name="weight", amount=0.5, n=2, dim=0)`
				- `# module.weight 为待剪枝的权重，amount 为要剪枝的比例`
				- `# n=2 表示采用 L2 norm，dim=0 表示剪枝的是第 0 维度`
			- 非结构化剪枝：
				- L1 范数挑稀疏元素置 0，掩码相乘；理论上 30 %→0 参数量，但未经专用稀疏库/硬件仍做乘法，**仅省存储不省算力**
				- PyTorch 一行：`torch.nn.utils.prune.l1_unstructured(module, name="weight", amount=0.3)`
				- `# module.weight 为待剪枝的权重，amount 为要剪枝的比例`
			- 实战：先非结构化探上限稀疏率，再按重要性转结构化，兼顾压缩比与加速比
		- 大模型剪枝代表工作
			- SparseGPT
				- 一次性非结构化剪枝 175 B OPT模型，60 % 权重置 0，仅用单卡 A100；通过稀疏行消元保持困惑度，推理需配套稀疏内核  
			- ShearedLLaMA
				- 结构化剪枝 + 继续预训练，把剪枝写成 **带约束的联合优化**  
				- 目标：$\min_{\theta,\mathbf z}\mathcal L_{\text{prune}}(\theta,\mathbf z,\lambda)=\mathcal L(\theta,\mathbf z)+\sum_i\lambda_i\mathcal L_i(\mathbf z_i-N_i^{\text{target}})$  
				- 掩码 $z∈{0,1}$ 控制头/维度/层数；用拉格朗日乘子动态平衡“剪枝率 vs 损失”  
				- 辅以**动态数据配比**：每 N 步按领域损失差（如实际与目标的PPL）调整采样权重，提升继续预训练效率
				- 结果：LLaMA-2 参数从 7 B → 2.7 B，50 B token 再训练后恢复 87.8 % 下游精度，吞吐↑2.3×
