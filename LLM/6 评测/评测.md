1. 评测指标与评测方法
	- （1）常见评测指标
		- 1）语言建模核心指标：困惑度 PPL
			- 定义
				- $\mathrm{PPL}(u)=\exp\!\bigl(-\frac{1}{T}\sum_{t=1}^T\log p(w_t\mid w_{<t})\bigr)=P(w)^{-1/T}$ 
				- 文本序列概率几何平均的倒数，值越小表示模型对文本预测越“确定”
			- 特性
				- 与交叉熵等价：$\mathrm{PPL}=2^{H(p,q)}$，直接反映分布差异
				- 长度无关：归一化到每个 token，可跨序列比较
				- 数值稳定性：实际实现用对数累加，避免长序列下溢
			- 使用注意
				- 依赖 tokenizer 一致；不同 vocab 的 PPL 不可比
				- 仅衡量“拟合度”，不保证生成质量或事实正确性
				- 对重复词、高频停顿敏感，需结合其他指标综合评估
		- 2）分类任务核心指标
			- 精确率
				- $\mathrm{Precision}=\frac{\mathrm{TP}}{\mathrm{TP}+\mathrm{FP}}$
				- 预测正例中实际正例的比例，衡量“查准”能力
			- 召回率
				- $\mathrm{Recall}=\frac{\mathrm{TP}}{\mathrm{TP}+\mathrm{FN}}$
				- 实际正例中预测正例的比例，衡量“查全”能力
			- $F_1$ 分数
				- $F_1=2\cdot\frac{\mathrm{Precision}\cdot\mathrm{Recall}}{\mathrm{Precision}+\mathrm{Recall}}$ 
				- 精确率与召回率的调和平均，综合性能，适用于类别分布相对均衡的场景
			- 宏平均 & 微平均：多分类时对各类别 $F_1$ 分别求平均（宏）或整体 TP/FP/FN 汇总后计算（微），用于处理类别不平衡。
		- 3）条件文本生成任务评测指标
			- BLEU
				- 计算候选文本与参考文本的𝑛元组（𝑛-gram）共现频率，通常𝑛=4
				- 公式：$\mathrm{BLEU}=BP\cdot\exp\Bigl(\sum_{n=1}^4 w_n\log P_n\Bigr)$，权重$w_n=\frac14$，
				- 𝑛元组的精确率：$P_n=\frac{\sum_{n\text{-gram}\in C}\min\!\bigl(\mathrm{count}_C,\,max_R\mathrm{count}_R\bigr)}{\sum_{n\text{-gram}\in C}\mathrm{count}_C}$  
				- 长度惩罚因子：$BP=\begin{cases}1&\text{if }l_C>l_R\\ \exp(1-l_R/l_C)&\text{otherwise}\end{cases}$  
				- $l_C$ 和 $𝑙_R$ 分别表示候选文本的长度和最短的参考文本长度
				- 特点：精度导向，高 n-gram 匹配奖励流畅；对同义词、语序变化敏感，偏低多样性
			- ROUGE-n
				- 主要侧重于召回率，即强调文本信息的覆盖度和完整性，通过计算𝑛元组上的召回率来评估候选文本的质量
				- 召回导向：$\mathrm{ROUGE\text{-}n}=\frac{\sum_{n\text{-gram}\in R}\min\!\bigl(\mathrm{count}_C,\,\mathrm{count}_R\bigr)}{\sum_{n\text{-gram}\in R}\mathrm{count}_R}$  
				- 其中，$C$ 代表候选文本，$R$ 代表参考文本
				- 用途：摘要任务更关注“覆盖参考要点”，故 ROUGE-1/2 成为事实性摘要主流指标
			- ROUGE-L
				- ROUGE变体，“L”代表最长公共子序列，不是简单计算固定长度的词组匹配，而是寻找候选文本和参考文本之间的最长公共子序列
				- 最长公共子序列 F1：$\mathrm{Recall}_{\text{LCS}}=\frac{|\mathrm{LCS}(C,R)|}{|R|},\;\mathrm{Precision}_{\text{LCS}}=\frac{|\mathrm{LCS}(C,R)|}{|C|}$
				- $LCS(𝐶,𝑅)$表示𝐶和𝑅之间的最长公共子序列长度
				- 加权 F1：$F_{\beta}=\frac{(1+\beta^2)\cdot P\cdot R}{\beta^2 P+R}$，通常 $\beta\rightarrow\infty$ 强调召回
				- 优势：允许非连续匹配，捕捉语序变化，适合长文本摘要与对话回复评估
		- 4）问答任务核心指标
			- 准确率：
				- $\mathrm{Accuracy}=\frac{\text{回答正确的样本数}}{\text{总样本数}}$ 
				- 宏观衡量整体正确率，适合单选题或“答案唯一”场景
			- 精确匹配率 EM
				- $\mathrm{EM}=\mathbb{I}(\text{预测答案} = \text{标准答案})$
				- 字符级完全匹配，过严但对事实型问答无模糊空间
			- F1（字符级）
				- $\mathrm{Precision}=\frac{|\text{预测}\cap\text{标准}|}{|\text{预测}|},\;\mathrm{Recall}=\frac{|\text{预测}\cap\text{标准}|}{|\text{标准}|},\;F_1=2\cdot\frac{P\cdot R}{P+R}$  
				- 允许答案片段重叠，适合抽取式/生成式混合场景（如SQuAD）
			- 数学推理：
				- 通过表达式等价性判断（sympy归一化后比较），避免数值格式差异导致的误判
		- 5）执行类任务指标
			- 成功率（Success Rate）
				- $\mathrm{SR} = \frac{\text{成功完成任务数}}{\text{总任务数}}$ 
				- 广泛用于智能体、工具调用、机器人控制等场景；成功判定依赖外部反馈（编译通过、测试用例全过、机械臂到达目标等）
			- Pass@k（代码合成（Code Synthesis）能力指标）
				- $\widehat{\mathrm{Pass}@k}=1-\frac{\binom{n-c}{k}}{\binom{n}{k}}$
				- 其中：$n$ 为同一问题生成的代码总数，$c$ 为通过测试符合要求的代码数
				- 计算：先无偏估计𝑘个代码全不符合要求的概率，再对其取反即得Pass@𝑘的无偏估计
				- 含义：随机抽 $k$ 个代码至少有一个通过的概率；组合估计避免全排列，采样 $n\approx 200$ 即可稳定收敛。
			- 用法提示
				- 仅报告 Pass@1 会低估能力，通常同时给出 Pass@1/10/100 曲线
				- 与 SR 区别：SR 是“一次就对”，Pass@k 允许模型“试错 $k$ 次”，更贴近实际 IDE 交互场景
		- 6）偏好排序核心指标：Elo 等级分（由物理学家Elo创建）
			- 期望胜率
				- $E_A = \frac{1}{1 + 10^{(r_B - r_A)/400}},\quad E_B = 1 - E_A$
				- 其中 $r_A, r_B$ 为当前 Elo 分，400 控制灵敏度
			- 动态更新
				- $r_A' = r_A + K(S_A - E_A)$
				- $S_A\in\{1,0.5,0\}$ 为实际得分（1表示胜利，0.5表示平局，0表示失败）
				- $K$ 越大对单点结果越敏感；常用 $32$（小样本）或 $16$（大样本）
			- 特点
				- 仅需成对比较，无需统一绝对标签，适合开放生成、主观质量评判
				- 收敛稳定后可得全局排序，且支持持续增量比较
				- 初始分通常设 1200；分差 200≈胜率 76 %，可直观解释模型间强弱
	- （2）评测范式与方法
		- 1）基础大模型评测流程（基于评测基准得方法）
			- 数据集选配
				- 知识：MMLU、C-Eval（多学科选择题）
				- 推理：GSM8K（数学）、BBH（Big-Bench Hard）、MATH（竞赛题）
				- 综合：OpenCompass 一键套件，覆盖上述子集
			- 提示构造
				- 零样本：仅“Problem:…Answer:”测纯泛化
				- 少样本：附 3-5 例带答案，激发 ICL 与推理；复杂题可加 “Let’s think step by step” 强制 CoT
			- 答案抽取
				- 规则脚本：正则匹配 “(A)”、“Answer: 42” 等固定模式
				- 容错：允许多格式（数字/字母/单词）归一化后再比对
			- 指标输出
				- 统一用 Accuracy；支持 per-domain 细拆
				- 排行榜（Open LLM Leaderboard）按多基准平均排名，提供可复现的 zero-shot/5-shot 双结果
		- 2）微调大模型评测双轨制
			- 基于人类
				- 成对比较：
					- Chatbot Arena 匿名对话 → 用户选优 → Elo 动态排名；
					- 优势：直观、噪声低；
					- 成本：众包+双边对话，周期长
				- 单一打分：
					- HELM 让评估员对摘要、安全、事实性逐项 Likert 评分；可细粒度诊断，但标度一致性难保证
			- 基于模型（Auto-Eval）
				- 方案：
					- 用 GPT-4 作“裁判”，对待测模型输出与参考输出做 side-by-side 胜/平/负；AlpacaEval 以此生成 win-rate
				- 改进：
					- 合成指令 + 多轮 MT-Bench，配合 CoT 提示提升 GPT-4 评判一致性（与人类 ρ>0.8）
				- 风险：
					- 闭源 API 限流、数据泄露、裁判模型偏好自身家族；
					- 开源替代：用人类评分蒸馏微调 Llama-3-judge，成本↓，一致性仍保持 ρ≈0.77
		- 3）三种评测方法对比
			- | 维度 | 基于基准 | 基于人工 | 基于模型 |
			  |---|---|---|---|
			  | 核心优势 | 自动化、可复现、低成本大规模 | 真实场景对齐、灵活、可细粒度 | 高效扩展、可解释、批量打分 |
			  | 主要劣势 | 数据污染、提示敏感、维度单一 | 主观偏差、成本高、不可重复 | 位置/冗长/自增强偏置、高难度任务能力不足 |
			  | 适用场景 | 基础能力监控、预训练阶段迭代 | 开放生成、对齐度、安全性 | 微调排行榜、快速筛选、众包前预过滤 |
			  | 改进方向 | 动态去污染、提示鲁棒性测试 | 评估员培训、多评估员一致性、混合打分 | 多裁判投票、反偏提示、开源裁判微调 |
2. 基础能力评测
	- （1）语言生成
		- 包括语言建模、条件文本生成、代码合成
		- 语言建模
			- 数据集：
				- Penn Treebank、WikiText-103、LAMBADA、The Pile
			- 指标：
				- 困惑度 $\mathrm{PPL}=\exp\!\bigl(-\frac{1}{T}\sum\log p(w_t|w_{<t})\bigr)$；
			- LAMBADA
				- 额外看“句末词预测准确率”以检验长距离依赖。
		- 条件文本生成
			- 机器翻译：
				- WMT14/16/21/22 多语言平行语料，自动用 BLEU、COMET；人工按 adequacy/fluency 打分
			- 文本摘要：
				- CNN/DailyMail、XSum，自动用 ROUGE-1/2/L；人工评信息量 & 流畅度
			- 长文本/结构化生成：
				- 探索性任务，已开始用 LLM-as-a-judge 缓解传统 n-gram 指标失效
		- 代码合成
			- 数据集：
				- HumanEval（164 题）、MBPP（500+ 题）、APPS（10k竞赛题）
			- 指标：
				- Pass@k 无偏估计，零样本生成函数体 → 单元测试全过即正确；
				- Codex 在 HumanEval 上 Pass@1 ≈ 28.8 %
				- AlphaCode 竞赛排名 Human Top 28 %
		- 条件文本生成两大痛点
			- 评测不可靠
				- 自动指标 BLEU/ROUGE 与人工打分常不一致；参考文本单一，无法捕捉语义等价变体
				- 人类评估：标注者间一致性低（κ<0.4），众包 vs 专家差异大
				- 新思路：用 LLM 生成多参考改写或无参考打分，但需校正顺序/冗长/自增强偏置
			- 专业化生成弱
				- 仅用网络语料，医疗、法律等术语密度不足，事实准确度↓
				- 领域继续预训练 → 灾难性遗忘：通用能力跌 10-30 %
				- 对齐税：RLHF 后下游任务性能再跌 5-15 %
				- 研究方向：参数高效适配（LoRA/adapter）、混合专家（MoE）路由、记忆外挂（检索增强）以保留通用性并注入专业知识
	- （2）知识利用
		- 划分为三个主要类别：闭卷问答、开卷问答、知识补全
		- 闭卷问答（Closed-book QA）
			- 数据集：
				- Natural Questions、WebQuestions、TriviaQA
			- 指标：
				- Exact Match、F1；模型仅依赖内部参数，无外部上下文
			- 趋势：
				- 参数量↑ → 准确率↑；同规模下继续预训练+问答语料增益最大
		- 开卷问答（Open-book QA）
			- 数据集：
				- OpenBookQA、SQuAD 1.1/2.0（含不可答）、Natural Questions（含段落）
			- 指标：
				- Accuracy（多项选择）、EM/F1（抽取）；
				- 需联合检索器（或搜索引擎），端到端微调可提升 5–10 %
			- SQuAD 2.0 额外考察“拒答”能力
		- 知识补全（Knowledge Completion）
			- 数据集：
				- FB15k-237（图谱链接预测）、WikiFact（句子级事实填空）
			- 指标：
				- Hits@k（图谱）、Accuracy@1（句子）；
				- 高频关系表现好，低频关系仍落后专用 KGE 模型
			- 指令微调显著缓解遗忘，InstructGPT 在 WikiFact 所有子集均优于GPT-3
		- 知识利用两大硬伤
			- 幻象（Hallucination）
				- 内在：输出与输入冲突；外在：无法验证其真伪
				- 检测：多采样一致性、对外检索冲突、不确定性阈值
				- 缓解：RLHF/RLAIF 对齐、检索增强（RAG）、自洽性投票；仍无法根治
			- 知识时效性
				- 参数冻结后知识即“快照”，新事实→过时答案
				- 重训成本大且易灾难性遗忘；增量训练需配回放/正则
				- 实用方案：
					- 外挂检索：实时搜索引擎或向量库，零参数更新即可“上网”
					- 参数编辑：ROME、MEMIT 等定位前馈层知识神经元，小范围改权重；大模型上成功率 <60 %，需继续研究高效、无副效应的编辑机制
	- （3）复杂推理
		- 复杂推理三大子类
			- 知识推理
				- 旨在考察模型通过逻辑关系与事实知识解决复杂任务的能力
			- 符号推理
				- 给定形式化的规则，让模型操作预定义好的符号以实现某些特定目标
			- 数学推理
				- 利用数学知识、逻辑推演与计算能力，解决数学问题或构建证明过程
			- | 子类 | 代表数据集 | 评测指标 | 关键挑战 |
			  |---|---|---|---|
			  | 知识推理 | CommonsenseQA、HellaSwag、SocialIQA | Accuracy | 需隐含常识，模型易生成“看似合理”但错误答案 |
			  | 符号推理 | Last-Letter Concat、Coin Flip | Accuracy | 上下文长度>训练示例即失效，需逐步显式提示 |
			  | 数学推理 | GSM8K、MATH | Answer Accuracy | GSM8K 2-步即可解；MATH 含几何/代数/概率，需 LaTeX 表达式严格等价验证 |
		- 提升配方
			- 思维链提示：零样本加“Let’s think step by step”即可↑
			- 继续预训练：在 60 B token 数学语料上训练，GSM8K Pass@1↑
			- 形式化辅助：把自然语言问题转成 Lean/Isabelle 代码，再借 ITP 交互证明，缓解单步逻辑跳错
		- 复杂推理两大痛点
			- 推理不一致
				- 现象：错误推理链→正确答案，或反之；提示微变即输出大变
				- 缓解：
					- 过程监督：给每步推理打分，用 ToT / Self-Refine 多路径自评
					- 代码形式化：把问题转可执行脚本，运行结果即最终答案，保证“过程-结果”一致
					- 多数投票：采样 N 条链，答案聚类后取高频，降低单条跳步错误
			- 数值计算拉胯
				- 根因：数字子词分词不一致（7_481 vs 748_15），训练集大数运算稀缺
				- 对策：
					- 数位分词：每 digit 独立 token，LLaMA 采用后4位 digit 运算准确率↑18 %
					- 合成算术微调：百万级表达式-结果对，训练 1 epoch 即可恢复多位乘除
					- 外挂计算器：提示调用 `calc()` API，把表达式替换成精确值再往后推理， GSM8K 错误率再降 30 %
3. 高级能力评测
	- （1）人类对齐
		- 人类对齐三维评估
			- | 维度 | 数据集 | 指标 | 关键看点 |
			  |---|---|---|---|
			  | 诚实性 | TruthfulQA (817 对抗题) | 人工 0-1 真实性得分 + 多选准确率 | GPT-4 仍有 27 % 虚假率 |
			  | 幻象检测 | HaluEval (35 k 陈述对) | Precision/Recall/F1 | 模型自检准确率≈70 %，低于人工 10 % |
			  | 社会偏见 | CrowS-Pairs (1.5 k 句子对) | 偏好偏见句比例 ↓ | 开源模型偏见率平均 58 %，高于随机 8 % |
			  | 性别-职业 | Winogender (720 句) | 代词消解准确率差异 | 男性职业代词↑6 %，显示隐性偏好 |
			  | 有害性 | RealToxicityPrompts (101 k 提示) | 毒性得分 0-1 ↓ | 经过滤模型毒性<0.05，未过滤可达 0.25 |
		- Chatbot Arena 评测体系
			- 众包 200 k+ 对战，Elo 实时榜；
			- 闭源模型（GPT-4、Claude-3）领先开源 80~120 Elo，差距主要体现于指令遵循与安全拒绝
	- （2）环境交互
		- 环境交互能力评测
			- | 场景 | 数据集 | 任务特征 | 核心指标 |
			  |---|---|---|---|
			  | 家庭具身 | ALFWorld（120 套住宅） | 纯文本观测→“go/take/put”序列，需长程规划与子目标管理 | 任务完成率（%） |
			  | 互联网 | WebShop（1.18 M 真实商品） | 根据 12 k 人类购物指令执行搜索-浏览-下单-结算全流程 | 平均产品得分 + 成功率 |
			  | 社会仿真 | Generative Agents 沙盒 | 25 个 LLM 代理 2 天社区生活，受控问答测记忆/规划，端到端观察关系与信息扩散 | 受控：问答准确率；端到端：关系网络密度、信息传播速度 |
		- 趋势
			- 把 LLM 当“策略网络”，用自然语言观测-动作接口替代传统 RL 的状态-动作空间，降低仿真门槛；
			- 多代理协作评估正从“单点问答”走向“集体行为度量”
	- （3）工具使用
		- 工具使用三维评估（搜索工具、模型工具、综合工具）
			- | 工具类型 | 数据集 | 任务要点 | 评测指标 |
			  |---|---|---|---|
			  | 搜索 | HotpotQA | 多跳问答需调用 Search API 整合≥2 篇 Wikipedia | EM/F1 |
			  | API | APIBench | 1 6450 条指令-API 对，覆盖 TorchHub /HuggingFace /TMDB | 1. 存在率 2. AST 结构匹配 3. 执行正确率 |
			  | 综合 | ToolBench | 16 k 真实 API（金融/电影/数学等）三级难度：单工具/同类多工具/跨类多工具 | 通过率（API 无报错）+ 胜率（与基线方案比较） |
			  | 自主生成 | 自研任务 | 模型先用自然语言写简易工具（Python 函数），再调用之完成主任务 | 工具语法可运行率 + 主任务成功率 |
		- 趋势：
			- 从“给定 API 列表”走向“自主发现/编写工具”，实现真正可扩展的智能体；
			- 上下文窗口有限催生 API 向量检索与压缩调用语法，降低描述 token 量
4. 公开综合评测体系
	- （1）MMLU
		- 规模：57 个子域（人文、社科、STEM、职业考试），共 13 908 四择题，难度横跨中学到专业资格
		- 协议：5-shot 提示，输出 A/B/C/D 概率，取 argmax 为答案
		- 指标：宏平均准确率（per-domain 等权）
		- 标杆：GPT-4 86.4 %；Llama-2-70B 68.0 %；国内 7B 模型普遍 45-55 %
		- 用法：预训练阶段快速扫盲，定位薄弱学科；可细拆“高中物理”“临床化学”等子分数指导继续预训练语料配比
	- （2）BIG-Bench
		- BIG-Bench 原版
			- 204 任务横跨语言学、儿童发展、数学、常识、物理、社偏、代码等；文本生成 + 四选一混合
			- 指标：BLEU/ROUGE/EM + 准确率 + Brier Score（置信度校准）
			- 结论：65 % 任务上，大模型（≥5 shot）超人类平均；剩余 35 % 仍落后，遂衍生 BBH
		- BIG-Bench Lite
			- 24 个高方差子集，开销≈1 GPU-hour，用于快速迭代监控
		- BBH（BIG-Bench Hard）
			- 23 个“人类水平 > 模型水平”任务，专测逻辑、因果、算术、词序敏感度；PaLM-540B 仅 56 %，人类 80 %
			- 用途：指导继续预训练数据配比与指令微调难度曲线
	- （3）HELM
		- 结构：
			- 自顶向下：先定 16 核心场景（任务 × 领域 × 语言），再选配套数据集；额外 26 扩展场景做深度切片
		- 7 维指标：
			- 准确性、校准性（Brier）、鲁棒性（对抗扰动）、公平性（不同群体性能差）、偏差（刻板关联）、有害性（毒性率）、效率（吞吐/能耗）
		- 覆盖：
			- Q&A、摘要、情感、毒性、信息检索等 42 子任务，一次性输出 100+ 分数雷达图
		- 用途：
			- 给模型做“体检报告”，可快速发现“高准确率但高毒性”或“校准不足”等隐性风险；当前已集成到开源库，支持一键复现
	- （4）C-Eval中文综合评测
		- 覆盖：
			- 52 学科、13 948 四选一，难度分初中/高中/大学/专业四级；附加 C-Eval Hard（高数/离散/大物/大化）专测推理上限
		- 协议：
			- 零样本 & 5-shot，支持 Direct / CoT 双提示；指标：平均准确率
		- 发现：  
			- 国产模型在“中国近现代史”“中医基础”等地缘学科领先 GPT 4 10~15 % 
			- STEM Hard 仍落后 20 %；CoT 提示平均增益 6 %，远低于英语 BBH 的 15 %，凸显中文数学语料稀缺
		- 用法：
			- 快速摸底中文知识盲区，指导继续预训练“中文百科 + 教材”配比；Hard 子集已单独拆出做难度锚点
	- （5）其他评测数据集与资源
		- 人类考试族：
			- AGIEval（中国高考+美国 SAT/GRE）、CMMLU（中文多学科）、MMCU（医学）、M3KE（中学知识）、Xiezhi（百科）：均四选一；
			- 闭源模型普遍领先开源 15-25 %，GPT-4 在 AGIEval 超人类平均但仍低于人类顶尖
		- 多语言/专项：
			- TyDiQA（跨语言抽取）、MGSM（多语数学）：测非英文能力
			- OpenLLM Leaderboard：持续集成 MMLU/BBH/HE/MT-Bench 等 6 项，一键提交 HF repo 即出雷达图
		- 开源框架：
			- Eleuther Harness：
				- 支持 200+ 任务，一行命令切换zero/5-shot、CoT、不同 tokenizer
			- OpenAI Evals：
				- YAML 模板定义新任务，与官方库兼容，社区已贡献100+自定义eval
		- 用法提示：
			- 先跑 Harness 得“基础分”→再用人类考试集看“中文/地域”差距→最后用 Arena/Elo 验证真实体验，三段式可快速定位模型短板

	- （7）评测代码实践
		- LLMBox：开箱即用的“大模型评测命令行工具箱”
			- 一键跑分：
				- 内置 53 个主流基准（MMLU、C-Eval、GSM8K、HumanEval 等）
				- 一条命令出结果
			- 模型全覆盖：
				- 既支持 HF 开源（Llama、Qwen、Mistral…）
				- 也支持 OpenAI/Claude API
			- 评测代码：
				- `pip install llmbox`
				- `python inference.py -m Qwen/Qwen1.5-72B -d cmmlu -shots 5`
				- 即可得到 5-shot 中文综合准确率
			- 加速友好：
				- 自动对接 vLLM、4-bit/8-bit 量化、前缀缓存，评测速度提升 3–10×
			- 结果直接可复现：
				- 输出含预测文件、指标、运行日志
				- 可一键提交 Open LLM Leaderboard
		- LLMBox 三模式速用
			- 基于候选文本的困惑度评测
				- 把每个选项与题干拼接，算 `PPL(option)`，取最小值；适合基座模型无指令格式
				- `python inference.py -m meta-llama/Llama-2-7b-hf -d hellaswag`
			- 基于候选项的概率评测
				- 一次性输入 `题干 + A/B/C/D`，比较字母 token 概率；适合对话模型，支持 0/5-shot
				- `python inference.py -m Qwen/Qwen1.5-72B -d cmmlu -shots 5`
			- 生成式评测
				- 自由生成答案，支持 CoT、温度、pass@k；可外挂 vLLM 加速与前缀缓存，一行实现量化 & 采样
				- `python inference.py -m microsoft/phi-2 -d gsm8k -shots 8 --load_in_4bit --temperature 0.1 --samples 100`
			- 输出：
				- 自动给出指标（Acc/EM/Pass@k）与详细预测文件
				- 可直接提交 Open LLM Leaderboard
